## Blocking gaps (must fix before v1.0)

**1. dulwich scaling cliff — RESOLVED (wave 15, commit ff13dfa)**
Going from pydantic (170 K LoC, 49 s, 1.3 GB) to dulwich (170 K LoC, 380 s, 8.5 GB) is a 7.7×
wall-clock and 6.5× memory blow-up for equal code size. The distinguishing feature is edge
density (1.80 edges/node in dulwich vs ~1.10 everywhere else), which points to a super-linear
cost in the translate/statespace/export stages. The generated GNN markdown is 485 K lines —
roughly 8× pydantic — so the markdown emitter is at minimum O(edges) rather than O(nodes).

This is a v1.0 blocker not because it crashes (it passes, exit 0), but because v1.0 will
be pointed at arbitrary Python repos. Django, SQLAlchemy, Celery, and any protocol-heavy
codebase will hit similar edge densities. A user who runs `cogant translate django` and waits
10+ minutes while their laptop's 16 GB of RAM fills up will file a bug, not a feature request.

Fix or bound it explicitly. Options in ascending effort:
- (a) Add a loud warning in docs and CLI when edge:node > 1.5 ("expect N GB RAM");
- (b) Profile and fix the O(edges) hot spot in `statespace` or the markdown emitter;
- (c) Introduce streaming/pagination in the GNN markdown emitter to cap file size.

Option (a) alone is not enough for a v1.0 claim of "handles arbitrary Python repos."
Option (b) or (c) is required, or scope the v1.0 claim to "repos with edge:node < 1.5."

**2. CI/CD pipeline — RESOLVED (wave 15)**
`.github/workflows/ci.yml` now enforces ruff, mypy --strict, pytest across Python 3.11–3.13,
rust-build, and pre-commit. The mypy step uses `--strict` (not `--ignore-missing-imports`).
This blocker is closed.

## Not blocking, but worth noting

**Native roundtrip evidence remains a stronger target**: The current v0.6 metrics
ledger does not report fresh role-preserved rows from the checked-in JSONL. It
reports 23 `STALE_LEGACY` rows, `role_preserved_count: 0`, and nullable native
score aggregates. Strict structural isomorphism remains a separate stronger
target, and semantic preservation beyond role populations remains future work.

**Dynamic enrichment requires user cooperation**: `--no-dynamic` is required for all real-world
repos unless the target repo's test suite was run under COGANT's coverage tracer first. The
implementation exists; the documentation and deployment story do not. This is a usability gap,
not a technical gap.

**viz/png_export.py coverage at 35%**: This module is 1273 lines of complex rendering code.
It drags the aggregate from ~84% to 81.5%. The module is optional (PNG export, not on the
hot path), but a 35% uncovered module is a reliability risk if that code path is exercised in
production. Consider either excluding it from coverage scope (with a `# pragma: no cover` or
`noqa` policy) or adding targeted tests for the core rendering paths.
